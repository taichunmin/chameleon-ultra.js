# TSDoc / typedoc conventions

API docs (the GitHub Pages site, `llms.txt`) are generated by `typedoc` (config: `typedoc.json`, entry points listed there — currently `src/index.ts`, `src/Crypto1.ts`, and each `src/plugin/*.ts` except `SlipEncoder.ts`/`BufferMockAdapter.ts`) plus `build-utils/typedoc.ts` for a post-pass. `**/*.spec.ts`/`**/*.test.ts` are excluded from the docs build. Don't strip TSDoc tags casually when editing exported code — they drive real output, not just editor tooltips:

`documents/*.md` files (e.g. `documents/demos.md`) are standalone doc pages, not code comments — they're picked up via `projectDocuments` in `typedoc.json` and rendered alongside the API reference. Each needs YAML frontmatter with `title`, `group` (top-level nav grouping, e.g. `Documents`), and `category` (e.g. `Guides`) to be placed correctly in the generated site's navigation.

- `@group <Name>` — assigns a method/property to a doc section (e.g. `Connection Related`, `Device Related`, `Slot Related`, `Reader/Writer Related`, `Emulator Related`, `Mifare Classic Related`, `Mifare Ultralight Related`, `DFU Related`, `Internal`). Section display order is fixed by `groupOrder` in `typedoc.json` — adding a genuinely new category requires adding it there too, or it falls into the catch-all `*` group at the end.
- `@internal` — marks a member as implementation detail; used throughout on private-ish helpers (`use()`, `addHook()`, `invokeHook()`, etc.) that are technically public (no `#` prefix, needed by plugins) but not meant for end users.
- `@hidden` — omits the member from generated docs entirely (stronger than `@internal`). Used sparingly, e.g. on `$adapter`.
- `@inline @expand` (always paired) — used on exported `interface`/`type` declarations in `src/types.ts` so typedoc inlines their shape at each use site instead of emitting a separate linked type page. Add this pair to any new shared type alias/interface meant to appear inline in method signatures rather than as its own doc page.
- `@param name - description` / `@returns description` — standard TSDoc; every public `cmdXxx()` method should document its params and return value this way, matching the existing methods in `src/ChameleonUltra.ts`.
- `@example` — used liberally, often with a `<details><summary>...` collapsible block for longer snippets (see the class-level doc comment on `ChameleonUltra` in `src/ChameleonUltra.ts`). Follow that pattern for substantial examples rather than inlining long code in the main doc line.
- `@see` / `{@link url | text}` — cross-reference external docs or related APIs.
- `@throws` — used rarely; add when a method's failure mode isn't obvious from its name/return type.

TSDoc syntax is lint-checked but only as a warning (`local-rules/tsdoc/syntax: 'warn'` in `.eslintrc.cjs`), so malformed tags won't fail CI — but they will render wrong on the docs site, so still fix them.
